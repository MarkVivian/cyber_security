                                    MEMORY AND ADDRESSING MODES.
                                ==================================
         THE FUNDAMENTALS OF MEMORY AND ADDRESSING MODES IN X86 ASSEMBLY.
    -------------------------------------------------------------------------
- The cpu is the only part of the computer that can actually do things(add, subtract, compare, etc).
- It can only operate on data that is stored in its registers.
- But the cpu only has a few registers (8 general purpose registers in x86), so it needs a way to access data that is stored in memory.

- RAM (your .DATA variables) is much larger than the register file, but it is also much slower to access.
- But the RAM is "dump" in the sense that it doesn't know how to do anything with the data it stores, it just holds it there until the cpu needs it.
    RULE : If you want to do something with data that is stored in memory, you have to first load it into a register, then you can operate on it with the cpu's instructions.

- Why move data from memory to registers?
    * Imagine you have a variable in your .DATA section that holds a number you want to add to another number. You can't directly add the value in memory to another value; you have to first load it into a register, perform the addition, and then if you want to store the result back in memory, you have to move it back from the register to memory.
        EG.
            section .DATA
            num1 DD 5
            num2 DD 10
            result DD 0

            section .TEXT
            global _start

            _start:
                mov RAX, [num1] ; Load the value of num1 (5) into RAX
                add RAX, [num2] ; Add the value of num2 (10) to RAX, now RAX = 15
                mov [result], RAX ; Store the result (15) back into memory at result

    * What is the difference between mov <mem>, <reg> and mov <reg>, <reg> like is the outcome the same?
        mov <mem>, <reg>
            mov [RBX], RAX 
                * Your telling the CPU " look at the number inside RBX, go to that memory address(location in RAM) and write the value of RAX to that memory address".
                * If RBX contains 0x00000000 and RAX contains 0x00000001, then after this instruction is executed, the value at memory address(location in RAM) 0x00000000 will be 0x00000001.
                * This actually saves the value of RAX to the memory address(location in RAM) of RBX, so that it can be accessed later on in the program.
        
            mov [var], RAX 
                * Your telling the CPU "look at the number inside RAX and copy that number into the memory address(location in RAM) of var".
                * If RAX contains 0x00000001, then after this instruction is executed, the value at memory address(location in RAM) of var will be 0x00000001.
                * This actually saves the value of RAX to the variable var in memory(RAM), so that it can be accessed later on in the program.
        
        mov <reg>, <reg>
            mov RAX, RBX 
                * Your telling the CPU "look at the number inside RBX and copy that number into RAX".
                * If RBX contains 0x00000001 and RAX contains 0x00000000, then after this instruction is executed, RAX will also contain 0x00000001.
                * No RAM is being accessed in this instruction, we are just copying the value of RBX into RAX. This is a much faster operation than accessing RAM since registers are much faster than RAM.

- Why move data from registers to memory?
    * After performing operations on data in registers, you may want to save the results back to memory for later use or to share with other parts of the program. For example, if you calculate a value that you want to print to the console or write to a file, you would need to move it from a register back to memory before you can use it for those purposes.


        I. DECLARING STATIC DATA REGIONS.
    --------------------------------------
- You can declare global variables (similar to static data regions) in x86 assembly using special assembler directives.
- Data declarations should be preceded by the `.DATA` directive.
- All the variables are mutable by default, but you can use the `CONST` directive to declare a variable as a constant (i.e. immutable).
- Following this directive, the directives:
    * `DB` (Define Byte) - declares a byte (8 bits) of data.
                         -a byte can store
                            * values from 0 to 255.
                            * a single ASCII character e.g. 'A' (which has the ASCII value of 65) or 'B' (which has the ASCII value of 66).
                                > Note: when declaring a byte with a character, the assembler will automatically convert the character to its corresponding ASCII value and store that value in the byte.
                        - in C : char 
                                
    * `DW` (Define Word) - declares a (2 byte) word (16 bits) of data.
                         - 2 bytes can store
                            * values from 0 to 65535.
                            * two ASCII characters e.g. 'AB' (which has the ASCII value of 65 for 'A' and 66 for 'B', so the word would store the combined value of 65 and 66).
                        - in C : short


    * `DD` (Define Doubleword) - declares a (4 byte) doubleword (32 bits) of data.
                            - 4 bytes can store
                                * values from 0 to 4294967295.
                                * four ASCII characters e.g. 'ABCD' (which has the ASCII value of 65 for 'A', 66 for 'B', 67 for 'C', and 68 for 'D', so the doubleword would store the combined value of 65, 66, 67, and 68).
                        - in C : int/long 

    * `DQ` (Define Quadword) - declares an 8 byte quadword (64 bits) of data.
                            - 8 bytes can store
                                * values from 0 to 18446744073709551615.
                                * eight ASCII characters e.g. 'ABCDEFGH' (which has the ASCII value of 65 for 'A', 66 for 'B', 67 for 'C', 68 for 'D', 69 for 'E', 70 for 'F', 71 for 'G', and 72 for 'H', so the quadword would store the combined value of those ASCII characters).
                         - in C : long long

- Declared locations can be labbeled with names for later reference - This is similar to declaring variables by name, but abides by some lower level rules.
    EG.
        # locations declared in sequence will be located in memory(RAM) next to one another.
        .DATA 
        var1 DB 64 : Declare a byte with the value 64 and label it var1. 
        
        var2 DB 'A' : Declare a byte with the value of the ASCII character 'A' (which is 65) and label it var2.
                    : the quotations ('') are mandatory when declaring a byte with a ASCII character.

        var3 DW BD : Declare a word with the value of the ASCII characters 'B' and 'D' (which are 66 and 68 respectively) and label it var3.
        var4 DD ABCD : Declare a doubleword with the value of the ASCII characters 'A', 'B', 'C', and 'D' (which are 65, 66, 67, and 68 respectively) and label it var4.

            DB 10 : Declare a byte with the value 10, but do not label it. This is useful for declaring data that does not need to be referenced later in the program.
                  : Its location is var4 + 1 since its in sequence after var4, but it cannot be referenced by name since it does not have a label.

In this case, we are using RAX to access the values of the doublewords in var7 because it is a convenient register for holding those values while we work with them in our program.

The choice of register is often a matter of convention and convenience, but as long as you are consistent in your use of registers and do not overwrite values that you need to keep, you can use any general-purpose register to access the values in var7.

To access the full 32-bit value of each doubleword, you need to use a register that is at least 32 bits wide, such as RAX, RBX, RCX, or RDX.
                            * If var7 was a single byte DB variable instead of a doubleword DD variable, then you could use AL to access its value since AL is 8 bits wide and can hold the value of a single byte. However, since var7 is declared as a doubleword, you need to use a 32-bit register to access its full value.

- what happens if i store a big word in a DW like elephant?
    * In most modern assemblers (like NASM), this will trigger an Error or Warning (e.g., "character constant too long"). The assembler will not silently discard the letters "e-p-h-a-n-t".

- why is DW called a word if it only stores 2 bytes?
    * The term "word" is a historical term that originated from older computer architectures where a word was typically 16 bits (2 bytes) in size. In modern x86 architecture, a word is still defined as 16 bits, even though the architecture itself is 32-bit or 64-bit. Therefore, the DW directive is used to declare a word (16 bits) of data, and it is called a word for historical reasons, even though it may not align with the current architecture's word size.

- what is the difference between using square brackets and without?
    EG.
        mov RAX, [RBX] # RAX = *EBX
            - [RBX] is taken as memory instead of register. Register <-> Memory
            - This tells the assembler to go to the address in memory/RAM of RBX and take the value stored at that address and put it into RAX. 

        mov RAX, RBX # RAX = RBX which is invalid due to size difference.
            - this is taken as register. Register <-> Register.
            - This tells the assembler to take the value stored in RBX and put it into RAX.
            EG.
                mov RBX, 0x12345678     ; RBX = 0x12345678
                mov [RBX], 0xABCDEF01 ; Memory at address RBX (0x12345678) = 0xABCDEF01

                mov RAX, RBX             ; RAX = 0x12345678 
                mov RAX, [RBX]           ; RAX = 0xABCDEF01 (value at memory address RBX)


    > Register <-> Memory (with brackets) : Size is set by the register, always works cleanly.
    > Register <-> Register (without brackets) : size must match exactly, use movzx/movsx if not.
        EG.
             mov EAX, AL ; THIS IS INVALID SINCE WE ARE MOVING 8 BIT DATA TO A 32 BIT REGISTER.
             
             ; CORRECT WAY.
             movzx EAX, AL  ; zero-extend : EAX = 0x000000xx
                            ; zero-extend always fills with 0's.
                            ; so if AL held "01000001" the value of EAX would be "00000000 00000000 00000000 01000001"
                            ; use when your value is unsigned (like a plain count, or ASCII code or just a positive number)

             movsx EAX, AL  ; Sign-extend : fills upper bits based on AL's sign bit.
                            ; fills with copies of the left most bit.
                            ; if AL held "01000001" the leftmost bit is "0" thus making EAX be "00000000 00000000 00000000 01000001"
                            ; if AL held "11001000" the leftmost bit is "1" thus making EAX be "11111111 11111111 11111111 11001000"
                            ; use when your value is signed (can be negative)

                > Use when moving from small to big.

    - All sub-registers share the same physical bits.
    EG.
        RAX = 0x 11 22 33 44 55 66 77 88  (all 64 bits = 8 bytes = 16 hex digits)
        EAX = 0x             55 66 77 88  (lower 32 bits only )
         AX = 0x                   77 88  (lower 16 bits only )
         AL = 0x                      88  (bits 0-7, lowest byte )
         AH = 0x                   77     (bits 8 - 15, second byte from bottom )

        ; Live example.
        mov rax, 0x112233445667788

        mov al, 0xFF    ; Change only the lowest byte.
        ; RAX is now 0x11223344556677FF (only AL changes)

        mov ah, 0x00    ; Change only bits 8-15
        ; RAX is now 0x11223344556600FF (only AH changes)

        mov ax, 0xBAEF   ; Change the lowest 16 bits (bits 0-15)
        ; RAX is now 0x112233445566BAEF (only AX changes)

        mov eax, 0xABCDEF01     ; Write to 32-bit EAX.
        ; RAX is now 0x00000000ABCDEF01 UPPER 32 BITS WERE ZEROED

            > The 32-bit zero-Extension trap 
                * Writing to a 32-bit register (like EAX) automatically zeroes the upper 32 bits of the full 64-bit register (RAX).
                * Writing to a 16-bit or 8-bit sub-register does NOT automatically clear the upper bits.
                    EG.
                        ; Scenario : RAX currently holds 0xAAAAAAAABBBBBBBB
                        
                        mov eax, 0x123455678   ; Write to 32-bit EAX.
                        ; RAX is now 0x0000000012345678 - Upper 32 bits were zeroed automatically.

                        mov ax, 0xBEEF      ; Write to 16-bit AX.
                        ; RAX is now 0x000000001234BEEF - only AX changed, nothing else zeroed.

                        mov al, 0xFF        ; Write to 8-bit AL 
                        ; RAX is now 0x000000001234BEFF - Only AL changed, nothing else zeroed.

            Why does the 2 byte part before AX have no name:
                > Its simply because it was never given one. 
                    * When Intel originally built the 8086 processor, the main register was AX (16 bits). Later when they expanded to 32 bits they added the upper 16 bits and called the full thing EAX (the E stands for Extended). But they never gave that upper portion its own name because there was no clean way to access it directly anyway
                    * You cannot say "give me just the upper 16 bits of EAX" with a single instruction. If you need them you have to shift the bits down:
                        EG.
                            mov EAX, 0xABCDEF10  ; EAX = 0xABCDEF10
                            mov EBX, EAX    ; copy EAX into EBX 
                            shr EBX, 16     ; shift right by 16 positions, upper bits drop into lower half
                                            ; EBX is now 0x0000ABCD, and the upper 16 bits of EBX are zeroed out.
                                            ; now BX contains what was the upper 16 bits of EAX
                            ```

                            In 64-bit mode this gets extended again — the full register becomes RAX (64 bits), and the upper 32 bits of RAX also have no name.


        II. ADDRESSING MEMORY.
        -----------------------
- In the examples above, where we used labels to refer to memory regions , these labels are actually replaced by the assembler with 32-bit quantities that specify addresses in memory.
- In addition to supporting referring to memory regions by labels (i.e. constant values), the x86 provides a flexible scheme for computing and referring to memory addresses: up to two of the 32-bit registers and a 32-bit signed constant can be added together to compute a memory address. One of the registers can be optionally pre-multiplied by 2, 4, or 8.
- The addressing modes can be used with many x86 instructions.
- Here we illustrate some examples using the mov instruction that moves data between registers and memory. This instruction has two operands: the first is the destination and the second specifies the source. 
    EG.
        mov EAX, [EBX] : Move the 4 bytes (32 bits) in memory at the address contained in EBX into EAX.
        
        mov [var], ecx : move the 4 bytes in ECX into the memory at the address specified by var.
        
        mov EAX, [EBX + 4] : Move the 4 bytes in memory at the address computed by adding 4 to the value in EBX into EAX.
        
        mov EAX, [EBX + ECX] : Move the 4 bytes in memory at the address computed by adding the values in EBX and ECX into EAX.
        
        mov EAX, [EBX + ECX*4] : Move the 4 bytes in memory at the address computed by adding the value in EBX to 4 times the value in ECX into EAX.
        
        mov EAX, [EBX + ECX*4 + 8] : Move the 4 bytes in memory at the address computed by adding the value in EBX to 4 times the value in ECX and then adding 8 into EAX.
    
    EG OF INVALID ADDRESS CALCULATIONS
        mov EAX, [EBX + ECX*3] : This is invalid because the x86 architecture only allows for scaling factors of 1, 2, 4, or 8 when using index registers in memory addressing. A scaling factor of 3 is not supported and would result in an error during assembly.
        
        mov eax, [ebx - ecx] : This is invalid because the x86 architecture does not support using subtraction in memory address calculations. Memory addresses must be calculated using addition of registers and constants, not subtraction.
        
        mov eax, [ebx + ecx + edx] : This is invalid because the x86 architecture only allows for up to two registers to be used in memory address calculations. Using three registers (EBX, ECX, and EDX) in the calculation exceeds this limit and would result in an error during assembly.


        III. SIZE DIRECTIVES.
        -------------------------
- In general, the intended size of the data item at a given memory address can be inferred from the assembly code instruction in which it is referenced.
    EG.
        * In all of the above instructions, the size of the memory regions could be inferred from the size of the register operand.
        * When we are loading a 32-bit register, the assembler could infer that the region of memory wee were referred to was 4 bytes wide.
        * when we are loading a value stored in a single byte register to memory, the assembler could infer thatwe wanted the address to refer to a single byte in memory.

- However, in some cases the size of a referred-to memory region is ambiguous. 
    EG.
        * Consider the instruciton mov [ebx], 2. 
            > Should the instruction move the value 2 into the single byte at address EBX 
            > should it move the 32-bit interger representation of 2 into the 4 byte starting at address EBX.

        * Since either is a valid possible interpratation, the assembler must be explicitly directed as to which is correct.
        * The size directives 
            > BYTE PTR 
            > WORD PTR 
            > DWORD PTR 
            > QWORD PTR

                - Serves this purpose, indicating sizes of 1,2 and 4 bytes respectively.
            
            EG.
                mov BYTE PTR [ebx], 2 : Move 2 into the single byte at the address stored in EBX.
                mov WORD PTR [ebx], 2 : Move the 16-bit integer representation of 2 into the 2 bytes starting at the address in EBX.
                mov DWORD PTR [ebx], 2 : move the 32-bit interger representation of 2 into the 4 byte starting at the address in EBX.
                mov QWORD PTR [ebx], 2 : move the 64-bit interger representation of 2 into the 8 byte starting at the address in EBX.

- size mismatches (EAX vs AL)
    EG.
        mov EAX, [EBX] : This is valid because the assembler can infer that we want to move a 32-bit value from memory into the 32-bit register EAX.

        mov AL, [EBX] : This is valid because the assembler can infer that we want to move a single byte from memory into the 8-bit register AL.

        mov AL, EBX : This is invalid because the assembler cannot infer whether we want to move a single byte from EBX into AL (which would require truncation) or if we want to move the full 32-bit value of EBX into AL (which would require zero-extension). The size mismatch between the source and destination operands creates ambiguity that the assembler cannot resolve without explicit size directives.

        mov EAX, BYTE PTR [EBX] : This is valid because the size directive BYTE PTR explicitly indicates that we want to move a single byte from memory into the 32-bit register EAX. The assembler will zero-extend the byte value to fill the entire 32 bits of EAX.

        mov AL, DWORD PTR [EBX] : This is valid because the size directive DWORD PTR explicitly indicates that we want to move a 32-bit value from memory into the 8-bit register AL. The assembler will truncate the 32-bit value to fit into the 8 bits of AL, which may result in data loss if the original value exceeds 255.
